# L1 배치 데이터를 믿을 수 있는 이유

## 핵심 질문

> **"L1 배치 데이터도 시퀀서가 제출한 건데, 이 데이터를 어떻게 믿는다는 거지?"**

이것이 바로 Optimistic Rollup의 **가장 중요한 핵심**입니다!

---

## 핵심 답변

**"L1 배치 데이터를 무조건 믿는 것이 아니라, 배치 데이터의 '유효성 규칙'과 '트랜잭션 서명'으로 검증합니다!"**

---

## 1. 오해와 진실

### 오해

```
┌─────────────────────────────────────────────────────────────┐
│              잘못된 이해                                      │
└─────────────────────────────────────────────────────────────┘

❌ "Sequencer가 L1에 아무 데이터나 쓸 수 있다"
❌ "L1에 있으면 무조건 맞는 데이터다"
❌ "Sequencer를 신뢰해야 한다"
```

### 진실

```
┌─────────────────────────────────────────────────────────────┐
│              올바른 이해                                      │
└─────────────────────────────────────────────────────────────┘

✅ Sequencer는 L1에 아무 데이터나 쓸 수 있습니다!
   → BUT, 잘못된 데이터는 재파생 시 자동으로 무시됩니다!

✅ L1에 있다고 무조건 맞는 것이 아닙니다!
   → BUT, 유효성 규칙을 통과한 것만 L2 상태에 반영됩니다!

✅ Sequencer를 신뢰하지 않습니다!
   → 각 트랜잭션의 서명을 검증합니다!
```

---

## 2. 배치 데이터의 구조

### A. 배치에 포함된 것

```
┌─────────────────────────────────────────────────────────────┐
│              L1 배치 데이터 구조                              │
└─────────────────────────────────────────────────────────────┘

Batch Data (L1 Calldata/Blob):
┌──────────────────────────────────────┐
│ Transaction 1:                        │
│   ├─ From: Alice                     │
│   ├─ To: Bob                         │
│   ├─ Value: 1 ETH                    │
│   ├─ Nonce: 5                        │
│   ├─ Gas: 21000                      │
│   └─ Signature: 0x1234...  ⭐️       │
│      (Alice의 개인키로 서명!)         │
├──────────────────────────────────────┤
│ Transaction 2:                        │
│   ├─ From: Bob                       │
│   ├─ To: Charlie                     │
│   ├─ Value: 0.5 ETH                  │
│   ├─ Nonce: 10                       │
│   ├─ Gas: 21000                      │
│   └─ Signature: 0x5678...  ⭐️       │
│      (Bob의 개인키로 서명!)           │
└──────────────────────────────────────┘
```

**핵심:**
- 각 트랜잭션은 **사용자의 개인키로 서명**됩니다
- Sequencer는 서명된 트랜잭션을 **수집**할 뿐입니다
- Sequencer는 서명을 **위조할 수 없습니다**!

### B. Sequencer가 할 수 있는 것 vs 없는 것

```
┌─────────────────────────────────────────────────────────────┐
│          Sequencer의 권한                                     │
└─────────────────────────────────────────────────────────────┘

✅ Sequencer가 할 수 있는 것:
├─ 사용자 트랜잭션을 수집
├─ 트랜잭션 순서 결정 (정렬)
├─ 배치에 포함/제외 결정
└─ L1에 배치 데이터 제출

❌ Sequencer가 할 수 없는 것:
├─ 트랜잭션 내용 변조
├─ 서명 위조
├─ 다른 사용자 명의로 트랜잭션 생성
└─ 잔액 없는 트랜잭션 강제 실행
```

## 3. 배치 데이터 검증 과정

### A. 재파생 시 유효성 검증

```
┌─────────────────────────────────────────────────────────────┐
│        Challenger의 배치 데이터 재파생 과정                   │
└─────────────────────────────────────────────────────────────┘

Step 1: L1에서 배치 가져오기
┌──────────────────────────────────────┐
│ L1 BatchInbox TX                      │
│ Calldata: [TX1, TX2, TX3, ...]       │
└──────────────────────────────────────┘
  ↓
Step 2: 배치 디코딩
┌──────────────────────────────────────┐
│ 압축 해제                             │
│ 트랜잭션 목록 추출                    │
└──────────────────────────────────────┘
  ↓
Step 3: ⭐️ 각 트랜잭션 유효성 검증
┌──────────────────────────────────────┐
│ For each Transaction:                 │
│                                       │
│ 1. 서명 검증                          │
│    ├─ Recover(signature) == From?    │
│    └─ NO → TX 무효! 건너뜀            │
│                                       │
│ 2. Nonce 검증                        │
│    ├─ TX.nonce == Account.nonce?     │
│    └─ NO → TX 무효! 건너뜀            │
│                                       │
│ 3. 잔액 검증                          │
│    ├─ Account.balance >= TX.value?   │
│    └─ NO → TX 무효! 건너뜀            │
│                                       │
│ 4. Gas 검증                          │
│    ├─ TX.gasLimit reasonable?        │
│    └─ NO → TX 무효! 건너뜀            │
│                                       │
│ 5. ✅ 모든 검증 통과                  │
│    └─ TX 실행!                       │
└──────────────────────────────────────┘
  ↓
Step 4: 유효한 TX만 실행
┌──────────────────────────────────────┐
│ 상태 전이 계산                        │
│ StateRoot 업데이트                    │
└──────────────────────────────────────┘
```

### B. 실제 코드 예시

```go
// op-node/rollup/derive (개념적)
func ProcessBatch(batch []Transaction, currentState State) (State, error) {
    newState := currentState.Copy()

    for _, tx := range batch {
        // ⭐️ 1. 서명 검증
        signer, err := RecoverSigner(tx)
        if err != nil || signer != tx.From {
            // 서명 검증 실패 → TX 무시!
            log.Warn("Invalid signature, skipping tx", "tx", tx.Hash())
            continue  // ⭐️ 다음 TX로!
        }

        // ⭐️ 2. Nonce 검증
        account := newState.GetAccount(tx.From)
        if account.Nonce != tx.Nonce {
            // Nonce 불일치 → TX 무시!
            log.Warn("Invalid nonce, skipping tx", "tx", tx.Hash())
            continue
        }

        // ⭐️ 3. 잔액 검증
        if account.Balance < tx.Value + tx.GasPrice * tx.GasLimit {
            // 잔액 부족 → TX 무시!
            log.Warn("Insufficient balance, skipping tx", "tx", tx.Hash())
            continue
        }

        // ⭐️ 4. 모든 검증 통과 → TX 실행
        newState, err = ApplyTransaction(newState, tx)
        if err != nil {
            // 실행 실패 (예: revert) → revert 처리
            // 하지만 nonce는 증가, gas는 소모됨
        }
    }

    return newState, nil
}
```

## 4. 공격 시나리오와 방어

### 시나리오 1: Sequencer가 위조 트랜잭션 제출

```
┌─────────────────────────────────────────────────────────────┐
│          공격: Sequencer가 Alice 명의로 위조 TX 제출          │
└─────────────────────────────────────────────────────────────┐

악의적 Sequencer:
  ↓
1. 위조 트랜잭션 생성
   ┌──────────────────────────────────────┐
   │ TX:                                   │
   │   From: Alice                        │
   │   To: Sequencer (자기 자신)           │
   │   Value: 1000 ETH  ← 도둑질!          │
   │   Signature: 0xFAKE...  ← 위조!      │
   └──────────────────────────────────────┘
  ↓
2. L1에 배치로 제출
   └─ BatchInbox TX로 게시
  ↓
3. Output Root 제출
   └─ "Alice: 0 ETH, Sequencer: 1000 ETH"

방어:
  ↓
Challenger 재파생:
  ↓
1. L1에서 배치 가져오기
   └─ [위조 TX 포함]
  ↓
2. 서명 검증
   ┌──────────────────────────────────────┐
   │ Recover(0xFAKE...) = ???            │
   │ Expected: Alice's address            │
   │ Actual: Invalid or wrong address!    │
   │ → ❌ 서명 검증 실패!                  │
   └──────────────────────────────────────┘
  ↓
3. TX 무시!
   └─ 위조 TX는 실행되지 않음
  ↓
4. Output Root 계산
   └─ "Alice: 1000 ETH" (변경 없음)
  ↓
5. 비교
   ├─ Sequencer: "Alice: 0 ETH"
   ├─ Challenger: "Alice: 1000 ETH"
   └─ 불일치! → Fault Proof 제출 → Challenger 승리!

결과:
  ✅ 위조 TX는 무시됨
  ✅ Alice의 돈은 안전
  ✅ Sequencer의 거짓 Output Root 무효화
  ✅ Sequencer bond 몰수
```

### 시나리오 2: Sequencer가 잔액 없는 TX 포함

```
┌─────────────────────────────────────────────────────────────┐
│          공격: Sequencer가 불가능한 TX 포함                   │
└─────────────────────────────────────────────────────────────┐

악의적 Sequencer:
  ↓
1. Alice가 실제로 서명한 TX 제출
   ┌──────────────────────────────────────┐
   │ TX: (Alice가 진짜 서명!)              │
   │   From: Alice                        │
   │   To: Bob                            │
   │   Value: 1000 ETH                    │
   │   Signature: 0xREAL... ✅            │
   │                                       │
   │ 문제: Alice 잔액 = 100 ETH만!         │
   └──────────────────────────────────────┘
  ↓
2. L1에 배치로 제출
  ↓
3. Output Root 제출
   └─ "Alice: -900 ETH, Bob: 1100 ETH"  ← 불가능!

방어:
  ↓
Challenger 재파생:
  ↓
1. 배치 가져오기
  ↓
2. 서명 검증
   └─ ✅ 통과 (Alice가 진짜 서명함)
  ↓
3. 잔액 검증
   ┌──────────────────────────────────────┐
   │ Alice.balance = 100 ETH              │
   │ TX.value = 1000 ETH                  │
   │ 100 < 1000 → ❌ 실패!                 │
   └──────────────────────────────────────┘
  ↓
4. TX 무시!
   └─ 잔액 부족으로 실행 안 됨
  ↓
5. Output Root 계산
   └─ "Alice: 100 ETH, Bob: 100 ETH" (변경 없음)
  ↓
6. 비교
   └─ 불일치! → Fault Proof → Challenger 승리!

결과:
  ✅ 불가능한 TX는 무시됨
  ✅ 올바른 상태만 반영
  ✅ Sequencer 패배
```

### 시나리오 3: Sequencer가 TX 순서 조작 (MEV)

```
┌─────────────────────────────────────────────────────────────┐
│          공격: Sequencer가 자신에게 유리하게 순서 조작        │
└─────────────────────────────────────────────────────────────┐

실제 상황:
  사용자들이 제출한 TX:
  ├─ TX1: Alice → Bob: 1 ETH
  ├─ TX2: Bob → DEX: Swap 10 ETH
  └─ TX3: Sequencer → DEX: Swap 100 ETH (front-run!)

Sequencer가 L1에 제출한 순서:
  ├─ TX3: Sequencer → DEX (먼저!)  ← 이득!
  ├─ TX2: Bob → DEX
  └─ TX1: Alice → Bob

문제:
  ❓ 이것도 유효한가?

답변:
  ✅ 유효합니다!

이유:
  ├─ 모든 TX가 올바른 서명
  ├─ 모든 TX가 유효한 nonce
  ├─ 모든 TX가 충분한 잔액
  └─ 순서만 다름

하지만:
  ⚠️ 이것은 Optimistic Rollup의 문제가 아니라
     "Centralized Sequencer"의 문제!

해결책:
  ├─ Decentralized Sequencer (OP Stack 로드맵)
  ├─ Fair ordering 메커니즘
  ├─ Encrypted mempool
  └─ Multiple competing sequencers

중요:
  ⭐️ 이것은 "유효성"의 문제가 아니라 "공정성"의 문제
  ⭐️ Fault Proof는 "유효성"만 검증
  ⭐️ MEV는 별도 메커니즘으로 해결
```

## 5. 트랜잭션 서명의 역할

### A. 서명이 보장하는 것

```
┌─────────────────────────────────────────────────────────────┐
│              트랜잭션 서명의 보장                             │
└─────────────────────────────────────────────────────────────┘

Alice의 서명 = Signature(TX, Alice_PrivateKey)

보장:
├─ ✅ TX는 Alice가 직접 승인함
├─ ✅ TX 내용이 변조되지 않음
├─ ✅ Sequencer가 임의로 생성할 수 없음
└─ ✅ 암호학적으로 검증 가능

검증:
Recover(Signature) = Alice_Address
   ↓
ECDSA 서명 회복
   ↓
Alice의 개인키로만 생성 가능
   ↓
Sequencer는 Alice의 개인키 없음
   ↓
위조 불가능!
```

### B. 서명 검증 과정

```go
// 서명 검증 (개념적)
func VerifyTransaction(tx Transaction) (bool, common.Address) {
    // 1. TX 해시 계산
    txHash := Hash(tx.From, tx.To, tx.Value, tx.Nonce, tx.GasLimit, ...)

    // 2. 서명에서 공개키 복구
    publicKey := RecoverPublicKey(txHash, tx.Signature)

    // 3. 공개키에서 주소 계산
    address := AddressFromPublicKey(publicKey)

    // 4. 주소 확인
    if address != tx.From {
        return false, address  // ❌ 위조!
    }

    return true, address  // ✅ 유효!
}
```

**핵심:**
- 서명 검증은 **수학적 증명**입니다
- Sequencer가 위조할 수 없습니다
- 암호학이 보장합니다!

## 6. 배치 유효성 규칙

### A. 규칙 목록

```
┌─────────────────────────────────────────────────────────────┐
│              배치 데이터 유효성 규칙                          │
└─────────────────────────────────────────────────────────────┘

각 트랜잭션은 다음을 통과해야 함:

1. 서명 검증
   └─ RecoverSigner(tx) == tx.From

2. Nonce 검증
   └─ tx.Nonce == account.Nonce

3. 잔액 검증
   └─ account.Balance >= tx.Value + tx.Gas

4. Gas Limit 검증
   └─ tx.GasLimit <= BlockGasLimit

5. 형식 검증
   └─ TX 인코딩이 올바른가

6. Replay 보호
   └─ ChainID 확인

실패 시:
├─ TX 무시
├─ 다음 TX로 이동
└─ 상태 변경 없음
```

### B. 검증 실패 시 처리

```
┌─────────────────────────────────────────────────────────────┐
│          유효하지 않은 TX의 처리                              │
└─────────────────────────────────────────────────────────────┘

Batch: [TX1, TX2_invalid, TX3, TX4_invalid, TX5]
  ↓
재파생:
  ├─ TX1: ✅ 실행
  ├─ TX2: ❌ 서명 실패 → 건너뜀
  ├─ TX3: ✅ 실행
  ├─ TX4: ❌ 잔액 부족 → 건너뜀
  └─ TX5: ✅ 실행

최종 상태:
  └─ TX1, TX3, TX5만 반영됨

⭐️ 핵심:
   - 배치 전체가 무효가 되지 않음
   - 유효한 TX만 선택적으로 실행
   - 결정론적 (모든 노드가 같은 결과)
```

## 7. 왜 이 모델이 안전한가?

### A. 보안 계층

```
┌─────────────────────────────────────────────────────────────┐
│              다층 방어 시스템                                 │
└─────────────────────────────────────────────────────────────┘

Layer 1: 암호학적 보장
├─ ECDSA 서명
├─ 개인키 소유권 증명
└─ 위조 불가능

Layer 2: 프로토콜 규칙
├─ Nonce 시스템
├─ 잔액 검증
└─ Gas 메커니즘

Layer 3: 재파생 검증
├─ 모든 노드가 독립적으로 검증
├─ 결정론적 실행
└─ 불일치 자동 감지

Layer 4: 경제적 보장
├─ Fault Proof 시스템
├─ Bond 메커니즘
└─ 인센티브 정렬
```

### B. 신뢰 가정

```
┌─────────────────────────────────────────────────────────────┐
│              신뢰하는 것 vs 신뢰하지 않는 것                  │
└─────────────────────────────────────────────────────────────┘

✅ 신뢰하는 것:
├─ 수학 (ECDSA, Hash)
├─ 암호학
├─ L1 이더리움의 불변성
├─ EVM의 결정론
└─ 최소 1명의 정직한 Challenger

❌ 신뢰하지 않는 것:
├─ Sequencer
├─ L2 RPC
├─ 다른 Challenger들
└─ 배치 데이터의 "정직성"
```

## 8. 결론

### 핵심 답변

```
┌─────────────────────────────────────────────────────────────┐
│              질문에 대한 최종 답변                            │
└─────────────────────────────────────────────────────────────┘

질문:
"L1 배치 데이터도 Sequencer가 제출한 건데,
 이 데이터를 어떻게 믿는다는 거지?"

답변:
┌──────────────────────────────────────┐
│ L1 배치 데이터를 "믿지" 않습니다!     │
│ 각 트랜잭션을 "검증"합니다!           │
└──────────────────────────────────────┘

검증 방법:
1. 트랜잭션 서명 검증
   └─ 사용자의 개인키로 서명되었는가?

2. Nonce 검증
   └─ 올바른 순서인가?

3. 잔액 검증
   └─ 충분한 잔액이 있는가?

4. 유효하지 않은 TX 무시
   └─ 위조, 잔액 부족 TX는 자동 제외

5. 유효한 TX만 실행
   └─ 결정론적으로 상태 계산

핵심:
├─ Sequencer를 신뢰하지 않음
├─ 각 TX의 암호학적 서명을 신뢰
├─ 사용자만이 자신의 TX를 승인 가능
└─ Sequencer는 TX를 수집할 뿐, 생성 못함
```

### 비유

```
┌─────────────────────────────────────────────────────────────┐
│              우편 배달부 비유                                 │
└─────────────────────────────────────────────────────────────┘

Sequencer = 우편 배달부
├─ 편지들을 수집
├─ 순서를 정함
└─ 배달함에 전달

하지만:
├─ ❌ 편지 내용을 마음대로 쓸 수 없음
├─ ❌ 다른 사람 이름으로 편지 쓸 수 없음
└─ ✅ 각 편지에는 서명이 있음!

검증:
├─ 받는 사람이 서명 확인
├─ 위조 편지는 무시
└─ 유효한 편지만 처리

결과:
├─ 배달부를 신뢰할 필요 없음
├─ 서명만 믿으면 됨
└─ 암호학이 보장!
```

### 핵심 원리

```
┌─────────────────────────────────────────────────────────────┐
│              Optimistic Rollup의 핵심 원리                    │
└─────────────────────────────────────────────────────────────┘

1. Sequencer는 TX를 "수집"할 뿐, "생성" 못함
   └─ 각 TX는 사용자가 서명

2. L1 배치 데이터는 "제안"일 뿐
   └─ 재파생 시 검증됨

3. 유효하지 않은 TX는 자동 필터링
   └─ 서명, nonce, 잔액 검증

4. 올바른 상태만 반영
   └─ 결정론적 실행

5. Output Root로 최종 검증
   └─ Fault Proof로 증명

결론:
"L1 배치 데이터"를 신뢰하는 것이 아니라,
"암호학적 서명"과 "유효성 규칙"을 신뢰합니다!
```
